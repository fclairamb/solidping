# Email — fix Gmail rendering and capture message-id in job output

## Context

Despite spec `2026-05-03-37` polishing the transactional templates, invitation
emails render as ASCII art in Gmail (nested-table boundaries, literal `**bold**`
markers, line-broken every ~40 chars). See screenshot attached to the original
report.

Diagnosis:

1. **Wrong MIME-alternative ordering.** `server/internal/email/sender.go:128-140`
   sets HTML first via `SetBodyString` and plaintext last via
   `AddAlternativeString`. Per RFC 2046 §5.1.4, `multipart/alternative` parts must
   be ordered from least to most preferred — preferred *last*. Spec-compliant
   readers (Gmail, Apple Mail) therefore pick our plaintext over our HTML.
2. **Auto-generated plaintext from a nested-table HTML layout.**
   `server/internal/email/formatter.go:91` calls `prem.TransformText()`, which
   produces lynx-style ASCII rendering of the wrapper/container/header/content
   tables in `templates/base.html`. That output is never going to be presentable.

Separately, the email job records only `{sent, message}` in `Job.Output`
(`server/internal/jobs/jobtypes/job_email.go:142-145`). The SMTP `Message-ID`
header — auto-generated by `go-mail` on each send — is discarded. Other parts
of the codebase already store provider IDs under `messageId`
(`server/internal/handlers/emailcheck/handler.go:245`); the email job should
follow the same convention so we can correlate inbox events back to the job
that produced them.

## Honest opinion

The "styling is broken" framing is misleading. The CSS *is* inlined; Gmail
just isn't looking at the HTML because the MIME order tells it to prefer text.
Two minimal changes — reverse the order, drop the auto-text — make every
modern client render the HTML we already designed.

We could instead replace `premailer.TransformText()` with hand-written `.txt`
templates per email, but that doubles the per-template maintenance for
marginal benefit. Most B2B SaaS sends HTML-only transactional mail without
deliverability problems; revisit if Mailpit/Postmark stats argue otherwise.

Reverting `base.html` to a div layout would also fix the auto-text quality,
but spec 37 deliberately chose tables for Outlook compatibility. Keep that
decision out of this spec.

## Scope

**In:**
- Reverse multipart ordering in `SMTPSender.setBody` (text first, HTML last).
- Drop the auto-generated plaintext alternative from
  `TemplateFormatter.Format`. Adjust the interface and callers.
- Capture the Message-ID after send and store it as `messageId` in
  `Job.Output` for email jobs.
- Adjust tests for the new behavior; add a regression test that asserts the
  wire-level MIME structure has HTML last.

**Out:**
- Hand-written per-template `.txt` files (deferred; HTML-only is sufficient
  for now).
- Switching `base.html` away from table layout (spec 37 just landed; revisit
  separately if needed).
- Surfacing `messageId` in the jobs admin UI (backend-only change here; the
  UI already renders arbitrary `Output` JSON).
- Localized templates / `.en.html` variants — separate, larger spec.

## Implementation

### 1. Reverse multipart ordering

`server/internal/email/sender.go:127-140`. The MIME spec wants the preferred
format last; today we have it first.

```go
// setBody sets the plaintext and/or HTML body on the message.
// Per RFC 2046, multipart/alternative parts must be ordered from least to
// most preferred (preferred last) — so plaintext goes first, HTML last.
func (s *SMTPSender) setBody(mailMsg *mail.Msg, msg *Message) {
    switch {
    case msg.HTML != "" && msg.Text != "":
        mailMsg.SetBodyString(mail.TypeTextPlain, msg.Text)
        mailMsg.AddAlternativeString(mail.TypeTextHTML, msg.HTML)
    case msg.HTML != "":
        mailMsg.SetBodyString(mail.TypeTextHTML, msg.HTML)
    case msg.Text != "":
        mailMsg.SetBodyString(mail.TypeTextPlain, msg.Text)
    }
}
```

### 2. Stop generating the auto-plaintext

`server/internal/email/formatter.go`. Drop the `TransformText()` call and the
plaintext return. Update the `Formatter` interface (declared alongside `Sender`
in `server/internal/email/sender.go`).

```go
// Format renders a template with the given data. Returns the rendered
// subject (from a {{define "subject"}} block, or "" when the template has
// none) and the HTML body with inlined CSS.
func (f *TemplateFormatter) Format(templateName string, data any) (subject, html string, err error)
```

Callers:

- `server/internal/jobs/jobtypes/job_email.go:173` — drop the `text` return
  variable, drop `msg.Text = text`.
- `server/internal/email/formatter_test.go` — remove plaintext assertions
  added in spec 37 (the `wantText` slices in `TestFormatter_TransactionalTemplates`).
- `server/internal/notifications/email.go` — does not call `Format`, no change.

After this, `email.Message.Text` is only set when callers explicitly supply
raw plaintext (the `EmailJobConfig.Text` path). That path stays supported.

### 3. Capture Message-ID in `SendResult`

`server/internal/email/sender.go`:

```go
type SendResult struct {
    Sent      bool   `json:"sent"`
    Message   string `json:"message"`
    MessageID string `json:"messageId,omitempty"`
}
```

In `sendMessage`, after `client.DialAndSendWithContext` succeeds, read the
header from the message that was just sent:

```go
return &SendResult{
    Sent:      true,
    Message:   "email sent successfully",
    MessageID: strings.Trim(mailMsg.GetMessageID(), "<>"),
}, nil
```

`go-mail` populates the `Message-ID` header automatically during send if the
caller hasn't set one. `GetMessageID()` returns the value as wrapped in `<>`;
strip the brackets so the JSON value is a plain id (matches the
`emailcheck` convention at `server/internal/handlers/emailcheck/handler.go:245`).

If the API surface differs in the version of `go-mail` we vendor, fall back
to reading `mailMsg.GetGenHeader(mail.HeaderMessageID)` and stripping
brackets. Verify against `vendor/` or `go.mod` before implementing.

### 4. Persist `messageId` in `Job.Output`

`server/internal/jobs/jobtypes/job_email.go:141-145`:

```go
output := models.JSONMap{
    "sent":    result.Sent,
    "message": result.Message,
}
if result.MessageID != "" {
    output["messageId"] = result.MessageID
}
jctx.Job.Output = output
```

Conditional insertion keeps the field absent when sending is disabled
(`EmailConfig.Enabled == false`, which returns `Sent: false` and no id).
