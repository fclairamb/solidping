# Fake API for Testing

## Purpose
Create a fake API endpoint that simulates various up/down states and response behaviors for testing monitoring and health check functionality.

## Endpoint
`GET /api/v1/fake`

## Query Parameters

### State Management
- **`period`** (default: `120`)
  - The period in seconds between each state change (up ↔ down)
  - The state toggles automatically based on the current time: `(currentTime / period) % 2`
  - Example: With `period=120`, the API alternates between up/down every 2 minutes

### Response Configuration
- **`format`** (default: `json`)
  - The format of the response body
  - Allowed values: `json`, `xml`, `text`
  - Example: `?format=xml` returns XML response

- **`delay`** (default: `0`)
  - Delay in milliseconds before responding
  - Useful for testing timeout handling
  - Example: `?delay=5000` waits 5 seconds before sending response

### Status Codes
- **`statusUp`** (default: `200`)
  - HTTP status code to return when the endpoint is in "up" state
  - Example: `?statusUp=201`

- **`statusDown`** (default: `500`)
  - HTTP status code to return when the endpoint is in "down" state
  - Example: `?statusDown=503`

### HTTP Method Validation
- **`supportedMethod`** (default: `GET`)
  - The only HTTP method that will succeed
  - Other methods return `405 Method Not Allowed`
  - Allowed values: `GET`, `POST`, `PUT`, `DELETE`, `PATCH`
  - Example: `?supportedMethod=POST` only accepts POST requests

### Authentication & Headers
- **`requiredHeader`** (default: none)
  - Header that must be present in the request
  - Format: `Header-Name=expected-value`
  - If missing or incorrect, returns `400 Bad Request`
  - Example: `?requiredHeader=X-API-Key=secret123`

- **`requiredAuth`** (default: none)
  - Basic authentication credentials required
  - Format: `username,password`
  - If missing or incorrect, returns `401 Unauthorized`
  - Example: `?requiredAuth=admin,pass123`

### Redirects
- **`redirectTo`** (default: none)
  - URL to redirect to when the endpoint is in "up" state
  - Uses the status code specified by `redirectStatus`
  - Example: `?redirectTo=https://example.com/health`

- **`redirectStatus`** (default: `302`)
  - HTTP status code for redirects (when `redirectTo` is set)
  - Common values: `301` (permanent), `302` (temporary), `307` (temporary, preserve method)
  - Example: `?redirectStatus=301`

### Response Headers & Cookies
- **`setCookie`** (default: none)
  - Cookie to set in the response when in "up" state
  - Format: `cookie_name=cookie_value`
  - Example: `?setCookie=session=abc123`

- **`setHeader`** (default: none)
  - Custom header to set in the response when in "up" state
  - Format: `Header-Name=value`
  - Example: `?setHeader=X-Custom-Id=12345`

### Slow Response Simulation
- **`slowResponse`** (default: none)
  - Simulates a slow streaming response with random data
  - Format: `iterations,bytes,delay_ms`
  - Parameters:
    - `iterations`: Number of chunks to send
    - `bytes`: Number of random bytes to generate per chunk
    - `delay_ms`: Delay in milliseconds between chunks
  - **Behavior**:
    - Forces response to text mode (ignores `format` parameter)
    - Generates random printable ASCII bytes for each chunk
    - Appends a carriage return (`\n`) after each chunk
    - Flushes the response buffer after each chunk
  - Useful for testing timeout handling and partial response scenarios
  - Example: `?slowResponse=10,20,1000` sends 10 chunks of 20 random bytes with 1 second between each

## Response Format Examples

### JSON (default)
```json
{
  "status": "up",
  "timestamp": "2026-01-02T10:00:00Z"
}
```

### XML
```xml

<response>
  <status>up</status>
  <timestamp>2026-01-02T10:00:00Z</timestamp>
</response>
```

### Text
```

timestamp: 2026-01-02T10:00:00Z
```

## Implementation Recommendations

### 1. State Calculation
- Use server's current Unix timestamp divided by period to determine state
- Formula: `isUp := (time.Now().Unix() / period) % 2 == 0`
- This ensures consistent state across multiple requests without storing state

### 2. Parameter Validation
- Validate all numeric parameters are within reasonable bounds
- `period`: minimum 1 second, maximum 86400 (24 hours)
- `delay`: maximum 30000ms (30 seconds)
- `slowResponse` iterations: maximum 100
- Return `400 Bad Request` for invalid parameters

### 3. Authentication Handling
- Check `requiredAuth` first (401 if fails)
- Then check `requiredHeader` (400 if fails)
- Then check `supportedMethod` (405 if fails)
- Process in this order to match standard HTTP error precedence

### 4. Response Priority
- If `redirectTo` is set and state is "up", redirect immediately (ignore other response settings)
- If `slowResponse` is set:
  - It takes precedence over `delay`
  - Forces text mode (ignores `format` parameter)
  - Still respects state-based status codes (`statusUp`/`statusDown`)
- Apply `setCookie` and `setHeader` only when state is "up"

### 5. Security Considerations
- **Don't** allow `redirectTo` to redirect to internal/private IPs
- Validate URLs to prevent SSRF attacks
- Limit response body size when using `slowResponse`
- Rate limit to prevent abuse

### 6. Logging
- Log each request with parameters used
- Include calculated state (up/down) in logs
- Useful for debugging test scenarios

## Use Cases

### Test periodic downtime
```
/api/v1/fake?period=60&statusDown=503
```
Alternates between 200 OK and 503 Service Unavailable every 60 seconds

### Test authentication requirements
```
/api/v1/fake?requiredAuth=user,pass&requiredHeader=X-API-Key=secret
```
Requires both Basic Auth and custom header

### Test slow/hanging endpoints
```
/api/v1/fake?delay=10000
```
Takes 10 seconds to respond (tests timeout handling)

### Test streaming/chunked responses
```
/api/v1/fake?slowResponse=5,100,2000
```
Sends response in 5 chunks of 100 random bytes (plus newline) with 2-second delays between chunks, forcing text mode output
