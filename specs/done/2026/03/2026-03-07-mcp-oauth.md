# MCP OAuth 2.1 Authentication

## Goal

Provide a generic OAuth 2.1 authorization server built into the application so that any OAuth-capable client (Claude Desktop, VS Code extensions, third-party tools) can authenticate without requiring users to manually configure custom headers or PAT tokens.

The current API only accepts PAT tokens or JWTs via `Authorization: Bearer`. OAuth adds a proper browser-based login flow on top of the existing user credentials, producing standard JWTs that work with all existing protected endpoints.

## How It Fits In

The existing auth stack is reused entirely:
- `auth.Service.Login()` validates email + password and issues a JWT
- `RequireAuth` middleware validates JWTs -- **no changes needed here**
- OAuth tokens ARE the existing JWT access tokens; the flow just produces one through a browser redirect instead of a direct API call

## New Endpoints

Four new routes:

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/.well-known/oauth-authorization-server` | OAuth server metadata (RFC 8414) |
| `POST` | `/api/v1/mcp/register` | Dynamic client registration |
| `GET/POST` | `/api/v1/mcp/authorize` | Login form + auth code issuance |
| `POST` | `/api/v1/mcp/token` | Exchange auth code for JWT access token |

Any protected endpoint can advertise the OAuth server via a `WWW-Authenticate` header on 401 responses, allowing clients to auto-discover the flow.

## Flow

```
1. Client connects to any protected endpoint (e.g. /api/v1/mcp)
2. Gets 401 with:
   WWW-Authenticate: Bearer resource_metadata="https://host/.well-known/oauth-authorization-server"
3. Fetches /.well-known/oauth-authorization-server
4. Registers client via POST /api/v1/mcp/register (if no client_id yet)
5. Opens browser to /api/v1/mcp/authorize?
     response_type=code
     &client_id=mcp_xxx
     &redirect_uri=http://127.0.0.1:PORT/callback
     &code_challenge=<S256>
     &code_challenge_method=S256
     &state=<random>
6. User logs in (existing login flow) and selects organization
7. Server validates credentials via auth.Service.Login()
8. Stores auth code -> {jwt, code_challenge, expiresAt} in memory
9. Redirects to redirect_uri?code=<code>&state=<state>
10. Client POSTs to /api/v1/mcp/token:
     code=<code>
     code_verifier=<verifier>
     grant_type=authorization_code
11. Server verifies PKCE, returns the JWT access token
12. Client uses Authorization: Bearer <jwt> for all subsequent API calls
    (Existing RequireAuth middleware handles this normally)
```

## Implementation

### New file: `internal/mcp/oauth.go`

Contains the OAuth handler methods:

```go
type OAuthHandler struct {
    auth    *auth.Service
    baseURL string // e.g. "https://app.solidping.com"
    codes   sync.Map // code(string) -> oauthCode
    clients sync.Map // clientID(string) -> oauthClient
}

type oauthCode struct {
    accessToken   string
    codeChallenge string
    redirectURI   string
    clientID      string
    expiresAt     time.Time // 5 min TTL
}

type oauthClient struct {
    clientID     string
    clientName   string
    redirectURIs []string
    createdAt    time.Time
}
```

**`HandleMetadata`** -- serves `/.well-known/oauth-authorization-server`:
```json
{
  "issuer": "<baseURL>",
  "authorization_endpoint": "<baseURL>/api/v1/mcp/authorize",
  "token_endpoint": "<baseURL>/api/v1/mcp/token",
  "registration_endpoint": "<baseURL>/api/v1/mcp/register",
  "response_types_supported": ["code"],
  "grant_types_supported": ["authorization_code"],
  "code_challenge_methods_supported": ["S256"],
  "token_endpoint_auth_methods_supported": ["none"]
}
```

**`HandleRegister`** -- `POST /api/v1/mcp/register`: Accepts `client_name` and `redirect_uris`, generates a `mcp_` prefixed client ID, stores in `clients` map. Returns `{ client_id, client_name, redirect_uris }`.

**`HandleAuthorize`** -- `GET`: render a minimal HTML login form passing through all query params as hidden fields. `POST`: validate email + password via `auth.Service.Login()`, generate a random hex code, store `oauthCode`, redirect to `redirect_uri?code=<code>&state=<state>`.

For `Login()`, the org code comes from the form (org selection dropdown). If the user belongs to a single org, it's auto-selected.

**`HandleToken`** -- validate `grant_type=authorization_code`, load+delete the code entry, verify PKCE (`BASE64URL(SHA256(code_verifier)) == code_challenge`), return:
```json
{
  "access_token": "<existing JWT>",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

### Modified: `internal/middleware/auth.go`

Add an `OAuthHint` middleware that sets the `WWW-Authenticate` header pointing to the OAuth discovery URL on any 401:

```go
func OAuthHint(publicURL string, next bunrouter.HandlerFunc) bunrouter.HandlerFunc {
    return func(w http.ResponseWriter, req bunrouter.Request) error {
        w.Header().Set("WWW-Authenticate",
            `Bearer resource_metadata="`+publicURL+`/.well-known/oauth-authorization-server"`)
        return next(w, req)
    }
}
```

Applied in `server.go` to the MCP endpoint:
```go
api.POST("/mcp", middleware.OAuthHint(cfg.PublicURL, authMiddleware.RequireAuth(mcpHandler.Handle)))
```

### Modified: `internal/app/server.go`

Register the new routes (no auth middleware needed):
```go
oauthHandler := mcp.NewOAuthHandler(s.services.Auth, cfg.PublicURL)

// OAuth metadata (well-known, outside /api/v1)
router.GET("/.well-known/oauth-authorization-server", oauthHandler.HandleMetadata)

// MCP OAuth endpoints
mcpGroup := api.NewGroup("/mcp")
mcpGroup.POST("/register", oauthHandler.HandleRegister)
mcpGroup.GET("/authorize", oauthHandler.HandleAuthorize)
mcpGroup.POST("/authorize", oauthHandler.HandleAuthorize)
mcpGroup.POST("/token", oauthHandler.HandleToken)
```

### Config

Uses `server.publicUrl` from config. Falls back to `http://localhost:4000` for dev. Used as `baseURL` in OAuth metadata.

## What Does NOT Change

- All existing API routes and handlers -- unchanged
- `RequireAuth` middleware -- unchanged (JWTs produced by OAuth are valid JWTs)
- `auth.Service` -- called read-only, no new methods needed
- No database migrations -- auth codes and clients are in-memory only (initially)
- PAT token auth still works -- nothing is removed

## Example: Claude Desktop

Once deployed, connecting Claude Desktop to the MCP server:
- **Remote MCP URL**: `https://app.solidping.com/api/v1/mcp`
- **OAuth Client ID**: leave empty (auto-registered)
- **OAuth Client Secret**: leave empty

Claude Desktop auto-discovers OAuth via the 401 hint, registers dynamically, opens a browser login page, and uses the resulting JWT for all requests. Any other OAuth 2.1-capable client works the same way.

## Security Notes

- PKCE with S256 is enforced -- no plain method accepted
- Auth codes expire in 5 minutes and are single-use (`LoadAndDelete`)
- No client secret validation (public client model -- Claude Desktop doesn't provide one)
- `redirect_uri` is validated against the registered client's URIs
- Access tokens are standard JWTs with the existing expiry (typically 1h)
- Localhost redirect URIs (`http://127.0.0.1:*`) are allowed per OAuth 2.1 spec for public clients

## Implementation Order

1. PKCE utilities (challenge/verifier validation)
2. OAuth metadata endpoint (`GET /.well-known/oauth-authorization-server`)
3. Dynamic client registration (`POST /api/v1/mcp/register`)
4. Authorization endpoint (`GET/POST /api/v1/mcp/authorize`) with login form
5. Token endpoint (`POST /api/v1/mcp/token`) with PKCE verification
6. `OAuthHint` middleware and MCP route wrapping
7. Route registration in `server.go`
8. Tests (full OAuth flow with testcontainers)

## Future Extensions

- **Refresh tokens** -- issue refresh tokens alongside access tokens for automatic renewal
- **Token revocation endpoint** -- `POST /api/v1/mcp/revoke` (RFC 7009)
- **Persistent clients** -- move `clients` from `sync.Map` to database table for multi-node
- **Persistent codes** -- move `codes` from `sync.Map` to database cache for multi-node
- **Scopes** -- fine-grained scopes beyond full access (e.g., `checks:read`, `checks:write`, `incidents:read`)
- **Client management UI** -- dashboard page to view and revoke authorized MCP clients
