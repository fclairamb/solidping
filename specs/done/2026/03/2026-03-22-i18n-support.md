# i18n Support for dash0 & status0

## Overview

Add internationalization (i18n) support across both frontend apps:
1. **status0** (public status page) — status page owners can set a default language; visitors can override via `?lang=xx` query parameter
2. **dash0** (admin dashboard) — user selects preferred language from the UI, persisted in localStorage

Start with English + French to validate the full pipeline end-to-end.

## Motivation

The entire UI is currently English-only with ~100+ hardcoded strings in dash0 and ~15 in status0. Public-facing status pages should speak the language of their audience. The admin dashboard should also be usable by non-English speakers, though this is lower priority than the public page.

## Current State

- **Zero i18n infrastructure** — no translation library, no locale files, no language detection
- **All strings hardcoded** in components: navigation labels, form labels, validation errors, status labels ("Operational", "Outage", "Degraded"), button text, aria-labels
- **date-fns** is already a dependency in dash0 (supports locale-aware formatting), not yet in status0
- **Zod** is a dependency but not actively used for form validation — current validation in `check-form.tsx` is manual
- **Backend** returns structured errors with `code`/`title`/`detail` — the `code` field can serve as a translation key

## 1. Library Choice: react-i18next

**Dependencies** (both apps):
- `i18next`
- `react-i18next`
- `i18next-browser-languagedetector`

**Why react-i18next over react-intl:**
- `useTranslation()` + `t()` API is more concise than `<FormattedMessage>` for a codebase that uses inline strings
- Native namespace support for organizing translations by feature area
- JSON translation files work out of the box

## 2. Translation File Structure

```
apps/dash0/src/locales/
  en/
    common.json        # Buttons, labels, status words, generic errors
    nav.json           # Sidebar, breadcrumbs, menu items
    checks.json        # Check form, types, intervals
    incidents.json
    statusPages.json
    auth.json          # Login, register, tokens
    settings.json      # Account, org, server settings
  fr/
    common.json
    nav.json
    ...

apps/status0/src/locales/
  en/
    status.json        # All status page strings (~15-20 keys)
  fr/
    status.json
```

**Conventions:**
- Flat camelCase keys: `"allSystemsOperational": "All Systems Operational"`
- Interpolation: `"checkCreated": "Check {{name}} created successfully"`
- Plurals: `"itemCount_one": "{{count}} item"`, `"itemCount_other": "{{count}} items"`
- English files are the source of truth

**No shared package** — the overlap between apps is ~5 strings (status labels). Duplicating those is simpler than adding monorepo workspace plumbing.

## 3. Initialization

Each app creates an `src/i18n.ts` file imported in `main.tsx`:

```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector';
import enCommon from './locales/en/common.json';
// ... other namespaces

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    resources: { en: { common: enCommon, /* ... */ } },
    defaultNS: 'common',
    fallbackLng: 'en',
    interpolation: { escapeValue: false }, // React already escapes
  });

export default i18n;
```

Components use:
```typescript
const { t } = useTranslation('checks');
// ...
<Label>{t('urlRequired')}</Label>
```

## 4. Language Selection: status0 (Public Pages)

**Resolution order:**
1. `?lang=fr` query parameter (explicit, shareable)
2. `language` field from the status page API response (admin-configured default)
3. Browser `navigator.language`
4. Fallback: `en`

**Why query parameter over URL path (`/fr/org/slug`)?**
- Current routes are `/$org/$slug` — adding a language segment would conflict with org parsing
- Query parameter doesn't break existing URLs and is easy for page owners to share

**Implementation:**
- Route components read `?lang=` from search params
- Call `i18n.changeLanguage()` with the resolved language
- `StatusPageView` uses `useTranslation()` as normal

---

## 5. Language Selection: dash0 (Admin Dashboard)

**Resolution order:**
1. `localStorage.getItem('solidping_language')`
2. Browser language detection (via `i18next-browser-languagedetector`)
3. Fallback: `en`

**UI:** Language selector dropdown in the user menu in `AppSidebar.tsx`, next to the existing theme toggle.

## 6. Database: Status Page Language Config

**Migration:** `20260322000002_status_page_language`

```sql
ALTER TABLE status_pages ADD COLUMN language varchar(10);
```

Nullable — `NULL` means "use browser detection". Values are BCP 47 tags: `en`, `fr`, `de`, etc.

**Model** (`internal/db/models/status_page.go`):
```go
Language *string `bun:"language" json:"language,omitempty"`
```

Exposed in the existing PATCH endpoint and returned in the public GET endpoint.

## 7. Backend Considerations

**Phase 1: Frontend-only translation.** The backend continues returning English. The frontend already maps error codes to display messages in `error-views.tsx` — these are translated client-side.

**Phase 2 (later): Error code as translation key.** The frontend can use `t(\`errors.${error.code}\`)` with the API `title` as fallback. The backend never needs to know the user's language.

## 8. Date/Time & Number Formatting

**Dates:** Use `date-fns` with locale-aware formatting. Create a helper:

```typescript
// src/lib/date.ts
import { format } from 'date-fns';

export function formatDate(date: Date, formatStr: string): string {
  return format(date, formatStr, { locale: getDateFnsLocale() });
}
```

Consolidate the two duplicate `formatRelativeTime` implementations (`incidents.index.tsx`, `account.tokens.tsx`) into this module using `date-fns/formatDistanceToNow`.

Add `date-fns` to status0 (not currently a dependency).

**Numbers:** Use `Intl.NumberFormat` directly:
```typescript
// src/lib/number.ts
export function formatPercent(value: number, lang: string): string {
  return new Intl.NumberFormat(lang, {
    style: 'percent',
    maximumFractionDigits: 3,
  }).format(value);
}
```
