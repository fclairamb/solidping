# Two-Factor Authentication (2FA/MFA)

## Overview

Add TOTP-based two-factor authentication for user accounts. Strengthens security for self-hosted deployments where the dashboard may be exposed to the internet. Supported by Uptime Kuma and most SaaS competitors.

## Goals

1. Allow users to enable TOTP-based 2FA on their account
2. Require 2FA code during login when enabled
3. Provide recovery codes for account recovery
4. Admin ability to see 2FA enrollment status (not reset for users)

## Data Model

```sql
ALTER TABLE users ADD COLUMN totp_secret TEXT;
ALTER TABLE users ADD COLUMN totp_enabled BOOLEAN NOT NULL DEFAULT FALSE;
```

| Field | Type | Description |

| `totp_enabled` | `bool` (default false) | Whether 2FA is active |

## API Changes

### Existing Endpoints Modified

#### `POST /api/v1/auth/login`

When 2FA is enabled, instead of returning full tokens, return a partial response:

```json
// Request (unchanged)
{ "org": "default", "email": "admin@solidping.com", "password": "solidpass" }

// Response when 2FA required (HTTP 200)
{
  "requires2FA": true,
  "tempToken": "eyJhbGci..."
}
```

The `tempToken` is a short-lived JWT (5 minutes) with claim `{"purpose": "2fa", "userUid": "...", "org": "..."}`. It cannot be used for regular API access.

When 2FA is not enabled, login behaves as today (returns `accessToken` + `refreshToken`).

#### `GET /api/v1/auth/me`

Add `totpEnabled` field to response:

```json
{
  "uid": "...",
  "email": "admin@solidping.com",
  "totpEnabled": true,
  // ... existing fields
}
```

### New Endpoints

All 2FA management endpoints require authentication (Bearer token). Login verification endpoints require the `tempToken`.

#### `POST /api/v1/auth/2fa/setup` (authenticated)

Generates a TOTP secret and returns the provisioning URI for QR code rendering.

```json
// Response (HTTP 200)
{
  "totpUri": "otpauth://totp/SolidPing:admin@solidping.com?secret=JBSWY3DPEHPK3PXP&issuer=SolidPing",
  "secret": "JBSWY3DPEHPK3PXP"
}
```

The secret is stored encrypted on the user but `totp_enabled` remains `false` until confirmed. Calling setup again overwrites any unconfirmed secret.

#### `POST /api/v1/auth/2fa/confirm` (authenticated)

Confirm 2FA setup by validating a TOTP code. On success, enables 2FA and returns recovery codes.

```json
// Request
{ "code": "123456" }

// Response (HTTP 200)
{
  "recoveryCodes": [
    "a1b2c3d4e5",
    "f6g7h8i9j0",
    "k1l2m3n4o5",
    "p6q7r8s9t0",
    "u1v2w3x4y5",
    "z6a7b8c9d0",
    "e1f2g3h4i5",
    "j6k7l8m9n0",
    "o1p2q3r4s5",
    "t6u7v8w9x0"
  ]
}
```

Error responses:
- `400 VALIDATION_ERROR` — missing or empty code
- `400 INVALID_2FA_CODE` — code does not match
- `409 CONFLICT` — 2FA already enabled

#### `POST /api/v1/auth/2fa/verify` (with tempToken)

Verify TOTP code during login flow. The `tempToken` is passed in the Authorization header.

```json
// Request
{ "code": "123456" }

// Response (HTTP 200) — same as normal login
{
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJhbGci...",
  "loginAction": "default"
}
```

Error responses:
- `400 INVALID_2FA_CODE` — code does not match
- `401 UNAUTHORIZED` — temp token expired or invalid

#### `POST /api/v1/auth/2fa/recovery` (with tempToken)

Use a recovery code instead of TOTP during login.

```json
// Request
{ "recoveryCode": "a1b2c3d4e5" }

// Response (HTTP 200) — same as normal login
{
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJhbGci...",
  "loginAction": "default"
}
```

The used recovery code is removed from the stored list. Error responses:
- `400 INVALID_RECOVERY_CODE` — code does not match any stored code
- `401 UNAUTHORIZED` — temp token expired or invalid

#### `DELETE /api/v1/auth/2fa` (authenticated)

Disable 2FA. Requires current TOTP code for confirmation.

```json
// Request
{ "code": "123456" }

// Response (HTTP 200)
{ "message": "2FA disabled" }
```

Error responses:
- `400 INVALID_2FA_CODE` — code does not match
- `400 VALIDATION_ERROR` — 2FA not enabled

### TOTP Library

Use `github.com/pquerna/otp/totp`:

```go
// Generate
key, _ := totp.Generate(totp.GenerateOpts{
    Issuer:      "SolidPing",
    AccountName: user.Email,
})

// Validate (±1 period for clock drift)
valid := totp.Validate(code, secret)
```

### Recovery Codes

- Generate 10 random 10-character alphanumeric codes
- Store bcrypt-hashed in `totp_recovery_codes` JSONB array
- On use: compare with bcrypt, remove matched hash from array
- Display format: `xxxxx-xxxxx` (with dash for readability, strip before comparison)

### Temp Token

- JWT with `{"purpose": "2fa", "sub": userUID, "org": orgSlug, "exp": now+5min}`
- Signed with same `JWTSecret`
- Auth middleware rejects tokens with `purpose` claim for regular API access
- 2FA verify/recovery endpoints validate `purpose == "2fa"`

## Implementation Notes

- Use `github.com/pquerna/otp` Go library for TOTP generation/validation
- TOTP secret encrypted at rest with AES-256-GCM (key from config or derived from JWT secret)
- Recovery codes: bcrypt-hashed, single-use, 10 codes generated on setup
- Time window: accept codes from current and previous 30-second window (clock drift tolerance)
- OAuth/SSO users: 2FA applies to local login only; OAuth providers handle their own MFA
- PAT (Personal Access Tokens): bypass 2FA (tokens are already long-lived secrets)
- Temp tokens for 2FA: 5-minute expiry, cannot be used for regular API access

### TOTP Code Generation in Tests

For E2E tests, generate valid TOTP codes client-side using the `otpauth` npm package:

```typescript
import * as OTPAuth from "otpauth";

function generateTOTP(secret: string): string {
  const totp = new OTPAuth.TOTP({
    secret: OTPAuth.Secret.fromBase32(secret),
    digits: 6,
    period: 30,
  });
  return totp.generate();
}
```
