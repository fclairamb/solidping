# MCP — register a small set of monitoring-shaped prompts

## Context

The MCP spec defines three primitives: tools (model-callable functions), resources (model-readable data), and **prompts** (named, parameterized prompt templates the user can invoke from their MCP client). Prompts are the most underused part of the spec — most servers ship with zero. They're also exactly the right primitive for an ops-shaped tool: they let the user type `/triage` or `/draft-status-update` in their MCP client and get a well-shaped prompt that already references the right tools and context.

Today (`server/internal/mcp/handler.go:132-153`), the MCP handler doesn't register a `prompts` capability and doesn't implement `prompts/list` or `prompts/get`. The audit flagged this as a Tier 3 gap — worth doing once Tier 1 and Tier 2 land.

## Honest opinion

Three prompts are enough for a v1. More than five and you're padding. The bar is: would a user, on day one of using the MCP server, *already* want to invoke this without having to think about it?

My picks:

1. **`triage_incident`** — args: `incidentUid` (or "active" / most recent). Generates a prompt that asks the LLM to review the incident and its events, recent results, and any correlated check failures, then return a triage summary.
2. **`summarize_org_health`** — no args. Generates a prompt that asks the LLM to inventory all checks, surface the currently-failing ones, list active incidents, and produce a one-paragraph health summary.
3. **`draft_status_update`** — args: `incidentUid`, optional `tone` (`technical`/`non-technical`). Generates a prompt that asks the LLM to draft a customer-facing status page update from the incident metadata.

These mirror the three Tier 1 use cases from the AI roadmap (incident narrative, "explain failure", health overview). Prompts are how those use cases become *one click* for the user instead of "remember the right phrasing".

I'd resist adding `create_check_from_description` as a prompt — that's a workflow that wants the *tools* (`list_check_types`, `get_check_type_samples`, `validate_check`) and is too open-ended to template usefully. The user can just ask in natural language and the LLM will chain the tools.

## Scope

**In:**
- Implement `prompts/list` and `prompts/get` in `server/internal/mcp/handler.go:132-153`.
- Add `Prompts *PromptsCap` to `ServerCaps` (`protocol.go:67-71`) and advertise it in `handleInitialize`.
- New file `server/internal/mcp/prompts.go` containing the three prompt definitions and rendering logic.
- Tests for: prompts/list returns 3, prompts/get returns rendered messages, missing required args → error.

**Out:**
- Dynamic prompt registration. Static is enough.
- Prompt arguments with autocompletion (`completion/complete` method). Defer.
- More than 3 prompts. Resist scope creep.
- Translation/i18n of prompts. English only for v1.

## Prompt definitions (`server/internal/mcp/prompts.go`)

All prompts return a single-message conversation seed of role `user`, with the rendered text in a `text` content block. The text is what the LLM client will present to the user to send to the model.

### 1. `triage_incident`

```
Description: "Review an incident and produce a triage summary. The model will
  fetch the incident, its event timeline, the affected check's recent results,
  and return a paragraph summarizing what's happening, what likely caused it,
  and what to check next."
Arguments:
  - incidentUid (required): "Incident UID to triage. Use list_incidents to
    discover one if you don't have it."
```

Rendered message text:

```
Triage incident {{incidentUid}}.

Steps:
1. Call get_incident with uid="{{incidentUid}}" and with="check,events" to get
   the incident, the underlying check, and the timeline of events.
2. Call diagnose_check with identifier=<the check from step 1> to get recent
   results across regions.
3. Produce a triage summary in 3-5 sentences:
   - What is failing (check name, regions, error message).
   - When it started and how long it's been failing.
   - The most likely cause based on the error output.
   - One concrete next step (e.g. "check the upstream DNS", "verify the cert
     hasn't expired", "see if a deploy rolled out around the start time").
   Do not pad with caveats. If the data is ambiguous, say so in one phrase.
```

### 2. `summarize_org_health`

```
Description: "Produce a one-paragraph status summary of the organization's
  monitoring posture: which checks are failing, which incidents are active,
  overall posture."
Arguments: []
```

Rendered message text:

```
Summarize the current health of this organization's monitoring.

Steps:
1. Call list_incidents with state="active" to get all open incidents.
2. Call list_checks to get the full check inventory.
3. Produce a one-paragraph summary:
   - How many checks total, how many currently failing.
   - The active incidents (by check name and how long they've been open).
   - One sentence on overall posture ("everything green", "one minor incident",
     "multiple regions degraded").
   No bullet lists. One paragraph. Be direct.
```

### 3. `draft_status_update`

```
Description: "Draft a customer-facing status page update for an incident. The
  output is a draft for human review, not auto-published."
Arguments:
  - incidentUid (required): "Incident UID."
  - tone (optional): "Audience tone. Allowed: technical, non-technical.
    Default non-technical."
```

Rendered message text:

```
Draft a status page update for incident {{incidentUid}}.

Tone: {{tone|default:"non-technical"}}.

Steps:
1. Call get_incident with uid="{{incidentUid}}" and with="check,events" for
   the incident metadata and timeline.
2. Write a draft status update with:
   - Title: <2-6 words, customer-facing> (e.g. "Investigating elevated API
     errors", "Resolved: brief outage on EU region").
   - Body: 2-4 sentences. State what's affected, what's known, what's being
     done. {{tone == "non-technical" ? "Avoid jargon (no \"5xx\", no \"DNS
     SOA\")." : "Technical detail is welcome."}}
   - Status: pick one — investigating | identified | monitoring | resolved.

Output the draft formatted as:

  TITLE: ...
  STATUS: ...
  BODY: ...

Do not publish — this is a draft for human review.
```

### Rendering

A simple `os.Expand`-like helper or `text/template` is overkill for three prompts. Inline `strings.ReplaceAll` per argument is fine:

```go
func renderTriageIncident(args map[string]string) (string, error) {
    incidentUID, ok := args["incidentUid"]
    if !ok || incidentUID == "" {
        return "", fmt.Errorf("incidentUid is required")
    }
    return strings.ReplaceAll(triageIncidentTemplate, "{{incidentUid}}", incidentUID), nil
}
```

Keep templates as raw string constants in `prompts.go`. If the count grows past 5, switch to `text/template`.
