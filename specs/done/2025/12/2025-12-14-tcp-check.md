# TCP Check Specification

## Overview
The TCP checker performs TCP connection tests to verify that network services are accepting connections on specified ports. This is useful for monitoring service availability, checking port accessibility, and optionally validating service responses.

## Implementation
- Based on Go's standard `net` package (Dialer)
- Checker type: `"tcp"`
- Package location: `/back/internal/checkers/checktcp/`

## Configuration

### Required Fields
- `host` (string) - Target hostname or IP address
- `port` (int) - TCP port number to connect to (1-65535)

### Optional Fields
- `timeout` (duration, default: 5s) - Maximum time to wait for connection and response
- `send_data` (string, optional) - Data to send after connection is established (useful for protocol handshakes)
- `expect_data` (string, optional) - Expected response data. If specified, check fails if response doesn't contain this string
- `tls` (bool, default: false) - Use TLS/SSL encrypted connection
- `tls_verify` (bool, default: true) - Verify TLS certificate (only applies if tls=true)
- `tls_server_name` (string, optional) - Server name for TLS SNI (Server Name Indication). If not specified, uses the host field

### IPv4/IPv6 Support
- Auto-detect based on host resolution
- Support both IPv4 and IPv6
- If host resolves to both, use any of them
- Future: Add `ip_version` config option to force IPv4/IPv6

## Validation Rules

### Config Validation (Validate method)
- `host` must not be empty
- `port` must be >= 1 and <= 65535
- `timeout` must be > 0 and <= 60s
- If `expect_data` is specified, `send_data` should also be specified (warning only)
- `tls_verify` only applies when `tls` is true
- `tls_server_name` only applies when `tls` is true

## Execution Behavior

### Success Criteria (StatusUp)
- TCP connection established successfully within timeout
- If `send_data` specified: data sent successfully
- If `expect_data` specified: received response contains expected string
- If `tls` enabled: TLS handshake completed successfully
- If `tls_verify` enabled: certificate validation passed

### Failure Criteria (StatusDown)
- Connection refused (service not listening on port)
- Connection established but expected data not received
- TLS handshake failed
- TLS certificate validation failed (when tls_verify=true)

### Timeout (StatusTimeout)
- Context deadline exceeded before connection established
- Connection established but response not received within timeout

### Error (StatusError)
- Unable to resolve hostname
- Network unreachable
- Invalid configuration
- Unexpected network errors

## Metrics

Returned in `Result.Metrics` (for time-series aggregation):
- `connection_time_ms` (float64) - Time to establish TCP connection in milliseconds
- `tls_handshake_time_ms` (float64) - Time to complete TLS handshake in milliseconds (if TLS enabled)
- `total_time_ms` (float64) - Total time for connection, send, and receive operations
- `bytes_sent` (int) - Number of bytes sent (if send_data specified)
- `bytes_received` (int) - Number of bytes received

## Output

Returned in `Result.Output` (for diagnostic information):
- `host` (string) - Resolved IP address that was connected to
- `port` (int) - Port number connected to
- `ip_version` (string) - "ipv4" or "ipv6"
- `tls_enabled` (bool) - Whether TLS was used
- `tls_version` (string) - TLS version negotiated (e.g., "TLS 1.3") if TLS enabled
- `tls_cipher_suite` (string) - Cipher suite used if TLS enabled
- `received_data` (string) - First 1KB of received data (for debugging)
- `error` (string) - Error message if check failed

## Implementation Notes

### Connection Process
1. Resolve hostname to IP address
2. Create TCP connection with timeout using `net.DialTimeout`
3. If TLS enabled, upgrade to TLS connection using `tls.Client`
4. If `send_data` specified, write data to connection
5. If `expect_data` specified, read response and validate
6. Close connection gracefully

### TLS/SSL Support
- Use `tls.Dial` or upgrade existing connection with `tls.Client`
- Support SNI (Server Name Indication) via `tls_server_name`
- Validate certificates by default (can be disabled with `tls_verify=false`)
- Record TLS version and cipher suite for diagnostics
- Handle TLS handshake errors separately from connection errors

### Data Exchange
- `send_data` supports plain text strings
- For binary protocols, support hex encoding (e.g., "0x48656c6c6f")
- Read response with reasonable buffer size (e.g., 4KB)
- Set read deadline to prevent hanging on partial responses
- Partial matches for `expect_data` (contains, not exact match)

### DNS Resolution
- Resolve hostname before connecting
- Cache resolved IP during single check execution
- Return error if resolution fails
- Include resolved IP in output for diagnostics

### Error Handling
- Distinguish between connection refused (StatusDown) and network errors (StatusError)
- Connection timeout should be StatusTimeout, not StatusError
- TLS certificate errors should be StatusDown (service issue, not checker error)
- Provide clear error messages for troubleshooting

## Examples

### Basic Port Check
```json
{
  "host": "example.com",
  "port": 80
}
```

### HTTPS Check with TLS
```json
{
  "host": "example.com",
  "port": 443,
  "tls": true
}
```

### SSH Service Check
```json
{
  "host": "server.example.com",
  "port": 22,
  "expect_data": "SSH-2.0",
  "timeout": "3s"
}
```

### HTTP Request Check
```json
{
  "host": "example.com",
  "port": 80,
  "send_data": "GET / HTTP/1.1\r\nHost: example.com\r\n\r\n",
  "expect_data": "200 OK"
}
```

### SMTP Server Check
```json
{
  "host": "mail.example.com",
  "port": 25,
  "expect_data": "220"
}
```

### Database Check with TLS
```json
{
  "host": "db.example.com",
  "port": 5432,
  "tls": true,
  "tls_verify": true,
  "timeout": "10s"
}
```

## Future Enhancements
- Support for SOCKS/HTTP proxy connections
- Binary data support with base64 encoding
- Multiple send/receive exchanges (conversation patterns)
- Connection pooling for repeated checks
- TCP keep-alive configuration
- SO_LINGER and other socket options
- STARTTLS support for protocols that upgrade to TLS
- Certificate expiration checking (similar to HTTP checks)
- Alert on TLS version/cipher suite weaknesses
