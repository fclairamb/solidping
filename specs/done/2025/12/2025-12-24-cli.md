# solidping CLI Specification

A command-line interface for managing and monitoring solidping instances.

## Overview
The solidping CLI provides commands for:
- Server health and version checks
- Managing health checks (add, list, remove)
- Viewing check results with filtering

## Quick Summary

**Commands to Implement:**
1. `sp auth login` - Login and obtain session token
2. `sp auth logout` - Logout and clear session token
3. `sp auth me` - Show current authenticated user info
4. `sp server health` - Check server health
5. `sp server version` - Get server version
6. `sp checks list` - List all checks
7. `sp checks add <url>` - Add a new check
8. `sp checks remove <uid|slug>` - Remove a check
9. `sp results list` - List check results with filtering

**Key Features:**
- Configuration stored in `~/.config/solidping/settings.json`
- Two auth methods: username/password or PAT
- Both JSON (`--json`) and human-readable table output
- Global flags: `--config`, `--url`, `--org`, `--json`, `--verbose`
- Proper exit codes and error handling
- Built with urfave/cli/v3 + koanf/v2 in Go 1.24+

## Technical Requirements

### Language & Framework
- **Language**: Go 1.24+
- **CLI Framework**: `github.com/urfave/cli/v3` for command structure
- **Configuration**: `github.com/knadh/koanf/v2` for configuration management
- **Table Output**: `github.com/jedib0t/go-pretty/v6/table` for table formatting
- **HTTP Client**: Standard `net/http` with proper timeout handling
- **Output**: Support both JSON (`--json`) and human-readable table formats

### Exit Codes
- `0` - Success
- `1` - General error (network, API error, etc.)
- `2` - Configuration error (missing/invalid config)
- `3` - Authentication error
- `4` - Not found error (check, result, etc.)
- `5` - Validation error (invalid parameters)

## Running the CLI

Use the `sp` command (short alias):
```bash
sp [command] [subcommand] [options]
```

Or the full binary name:
```bash
solidping [command] [subcommand] [options]
```

### Global Flags
These flags are available for all commands:

- `--config <path>` - Path to config file (default: `~/.config/solidping/settings.json`)
- `--url <url>` - Override server URL from config
- `--org <name>` - Organization name (overrides config value, default: `default`)
- `--json` - Output in JSON format instead of human-readable tables
- `--verbose` / `-v` - Enable verbose logging (show HTTP requests/responses)
- `--help` / `-h` - Show help for command
- `--version` - Show CLI version

**Organization Resolution:**
1. If `--org` flag is provided, use that value
2. Otherwise, if `org` is set in config file, use that value
3. Otherwise, use `"default"`

## Configuration

### Location
Configuration is stored at: `~/.config/solidping/settings.json`

### Configuration Loading
1. Check if `--config` flag is provided, use that path
2. Otherwise, check `~/.config/solidping/settings.json`
3. If no config file exists, create it with default configuration
4. Ensure directory `~/.config/solidping/` exists (create with `0755` permissions)
5. Create config file with `0600` permissions (readable only by user)

### Token Storage
When using `sp auth login`, store the JWT token separately:
- **Token file location**: `~/.config/solidping/token`
- **Permissions**: `0600` (user read/write only)
- **Format**: Plain text containing just the JWT token
- Token takes precedence over username/password in config
- Use `sp auth logout` to remove the token file

### Authentication Methods

The CLI supports multiple authentication methods, checked in this order:

**Priority 1: Stored JWT Token** (from `sp auth login`)
- Location: `~/.config/solidping/token`
- Used automatically if file exists
- Cleared with `sp auth logout`

**Priority 2: Personal Access Token (PAT)** (from config)
```json
{
  "url": "http://localhost:4000",
  "auth": {
    "pat": "your_token_here"
  }
}
```
- Use PAT directly in `Authorization: Bearer <pat>` header
- No login call needed
- Long-lived token for automation

**Priority 3: Username and Password** (from config)
```json
{
  "url": "http://localhost:4000",
  "auth": {
    "username": "your_username_here",
    "password": "your_password_here"
  }
}
```
- Auto-login on first command
- JWT token stored to `~/.config/solidping/token`
- Token reused for subsequent commands

**Priority 4: Interactive Login**
- If no auth configured, prompt user to run `sp auth login`
- Exit with code `3` (authentication error)

### Default Configuration
When no config file exists, create one with these defaults:
```json
{
  "url": "http://localhost:4000",
  "org": "default",
  "auth": {
    "username": "admin",
    "password": "admin"
  }
}
```

**Configuration Fields:**
- `url` - Server URL (required)
- `org` - Organization name (optional, defaults to `"default"` if not specified)
- `auth` - Authentication configuration (required)

### Configuration Validation
- Validate `url` is a valid HTTP/HTTPS URL
- Ensure either `username`+`password` OR `pat` is present (not both)
- If validation fails, exit with code `2` and helpful error message

## Implementation

### Package Location
The CLI implementation should be located in: `back/pkg/cli`

### Binary Build
Create a separate main entry point in `back/cmd/sp/main.go`:
```go
package main

import (
    "context"
    "log"
    "os"

    "github.com/fclairamb/solidping/back/pkg/cli"
    "github.com/urfave/cli/v3"
)

func main() {
    app := cli.NewApp()
    app.Name = "sp"
    app.Usage = "solidping CLI - manage and monitor solidping instances"
    app.Version = "1.0.0" // TODO: Get from build info

    // Set up commands from cli package
    app.Commands = cli.GetCommands()
    app.Flags = cli.GetGlobalFlags()

    if err := app.Run(context.Background(), os.Args); err != nil {
        log.Fatal(err)
    }
}
```

Add to `Makefile`:
```makefile
build-cli:
	go build -o bin/sp ./back/cmd/sp

install-cli:
	go install ./back/cmd/sp
```

The binary should be named `sp` and optionally create a `solidping` symlink.

## Commands

### Authentication Commands

#### Login
Authenticate with username and password to obtain a session token:
```bash
sp auth login
```

The command will prompt interactively for credentials:
```
Username: admin
Password: ********
```

Or provide credentials via flags:
```bash
sp auth login --username admin --password admin
```

Or via environment variables:
```bash
SOLIDPING_USERNAME=admin SOLIDPING_PASSWORD=admin sp auth login
```

**API Endpoint**: `POST /api/v1/orgs/{org}/auth/login`
**Authentication**: Not required (this obtains the token)

**Request Body**:
```json
{
  "username": "admin",
  "password": "admin"
}
```

**Response**:
```json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user": {
    "uid": "user_123",
    "email": "admin@example.com",
    "name": "Admin User"
  }
}
```

**Token Storage**: Save the JWT token to `~/.config/solidping/token` with `0600` permissions

**Output (human-readable)**:
```
Login successful!
Logged in as: Admin User (admin@example.com)
Token saved to: ~/.config/solidping/token
```

**Output (--json)**:
```json
{
  "success": true,
  "user": {
    "uid": "user_123",
    "email": "admin@example.com",
    "name": "Admin User"
  },
  "token_path": "~/.config/solidping/token"
}
```

**Exit Codes**:
- `0` - Login successful
- `3` - Authentication failed (invalid credentials)
- `1` - Server error or network error

---

#### Logout
Clear the stored session token:
```bash
sp auth logout
```

**Implementation**:
1. Call `POST /api/v1/orgs/{org}/auth/logout` to invalidate server-side session (optional)
2. Delete `~/.config/solidping/token` file

**API Endpoint**: `POST /api/v1/orgs/{org}/auth/logout` (optional call)
**Authentication**: Required (uses current token)

**Output (human-readable)**:
```
Logout successful!
Token removed from: ~/.config/solidping/token
```

**Output (--json)**:
```json
{
  "success": true,
  "message": "Logged out successfully"
}
```

**Exit Codes**:
- `0` - Logout successful (even if no token exists)
- `1` - Error removing token file

**Note**: If no token file exists, still return success with message "Already logged out"

#### Add a Check
Add a new HTTP health check (URL required):
```bash
sp checks add https://one.one.one.one
```

With optional parameters:
```bash
sp checks add https://one.one.one.one \
  --type http \
  --interval 5s \
  --timeout 2s \
  --name "My Check" \
  --slug "http-cloudflare"
```

**API Endpoint**: `POST /api/v1/orgs/{org}/checks`
**Authentication**: Required

**Parameters:**
- `--type` - Check type (default: `http`)
- `--interval` - Check interval (e.g., `5s`, `1m`, default: `30s`)
- `--timeout` - Request timeout (e.g., `2s`, `5s`, default: `5s`)
- `--name` - Human-readable name (default: derived from URL)
- `--slug` - Unique identifier (default: auto-generated from URL)

**Request Body**:
```json
{
  "type": "http",
  "url": "https://one.one.one.one",
  "interval": "5s",
  "timeout": "2s",
  "name": "My Check",
  "slug": "http-cloudflare"
}
```

**Output (human-readable)**:
```
Check created successfully:
  UID: check_abc123
  Slug: http-cloudflare
  Name: My Check
  URL: https://one.one.one.one
```

**Output (--json)**:
```json
{
  "uid": "check_abc123",
  "slug": "http-cloudflare",
  "name": "My Check",
  "type": "http",
  "url": "https://one.one.one.one",
  "interval": "5s",
  "timeout": "2s"
```

**Exit Codes**:
- `0` - Check created successfully
- `3` - Authentication failed
- `5` - Validation error (invalid URL, interval, etc.)

#### Remove a Check
Remove a check by slug or UID:
```bash

```
Or:
```bash

```

**API Endpoint**: `DELETE /api/v1/orgs/{org}/checks/{uid}`
**Authentication**: Required

**Implementation Note**: If the argument doesn't start with `check_`, treat it as a slug and call `GET /api/v1/orgs/{org}/checks` to find the UID first.

**Output (human-readable)**:
```
Check removed successfully: http-cloudflare
```

**Output (--json)**:
```json
{
  "message": "Check removed successfully",
  "slug": "http-cloudflare"
}
```

**Exit Codes**:
- `0` - Check removed successfully
- `3` - Authentication failed
- `4` - Check not found

#### Filter Results
Filter results with multiple options:
```bash
sp results list \
  --status up \
  --since 2023-01-01T00:00:00Z \
  --check-slug http-cloudflare \
  --page 1 \
  --per-page 10 \
  --period-type raw \
  --region us
```

**API Endpoint**: `GET /api/v1/orgs/{org}/results?status={status}&since={since}&check_uid={uid}&page={page}&per_page={per_page}&period_type={type}&region={region}`
**Authentication**: Required

**Filter Options:**
- `--status` - Filter by status (`up`, `down`)
- `--since` - Results after this timestamp (ISO 8601 format)
- `--check-slug` - Filter by check slug (converted to UID via lookup)
- `--check-uid` - Filter by check UID directly
- `--page` - Page number for pagination (default: `1`)
- `--per-page` - Results per page (default: `100`, max: `1000`)
- `--period-type` - Aggregation period: `raw`, `hour`, `day`, `month`, `year` (default: `raw`)
- `--region` - Filter by region code

**Implementation Notes:**
- If `--check-slug` is provided, first look up the check UID via `GET /api/v1/orgs/{org}/checks`
- Convert all filter parameters to query string parameters
- Handle pagination metadata in output
- For human-readable output, show pagination info at bottom: `Showing 1-10 of 1523 results`

**Exit Codes**:
- `0` - Results retrieved successfully
- `3` - Authentication failed
- `4` - Check not found (if filtering by slug/uid)
- `5` - Validation error (invalid date format, etc.)

## Error Handling

### API Error Response Format
All API errors follow this format (from `base.HandlerBase`):
```json
{
  "title": "Human-readable error message",
  "code": "ERROR_CODE",
  "detail": "Detailed explanation of the error"
}
```

### Error Code Mapping
Map API error codes to CLI exit codes:

| API Error Code | CLI Exit Code | Description |
|----------------|---------------|-------------|
| `INTERNAL_ERROR` | 1 | Internal server error |
| `VALIDATION_ERROR` | 5 | Input validation failed |
| `NOT_FOUND` | 4 | Resource not found |
| `UNAUTHORIZED` | 3 | Authentication required |
| `FORBIDDEN` | 3 | Permission denied |
| `CONFLICT` | 5 | Resource conflict |
| `CHECK_NOT_FOUND` | 4 | Check does not exist |
| `ORGANIZATION_NOT_FOUND` | 2 | Organization does not exist |
| `USER_NOT_FOUND` | 3 | User does not exist |

### Error Display
**Human-readable mode:**
```
Error: Check not found
The check 'http-cloudflare' does not exist in organization 'default'.
```

**JSON mode:**
```json
{
  "error": {
    "title": "Check not found",
    "code": "CHECK_NOT_FOUND",
    "detail": "The check 'http-cloudflare' does not exist in organization 'default'."
  }
}
```

### Network Errors
- Connection refused: Exit code `1`, message: "Cannot connect to server at {url}"
- Timeout: Exit code `1`, message: "Request timed out after {duration}"
- DNS resolution failed: Exit code `1`, message: "Cannot resolve hostname {host}"

### API Helper Package (`apihelper/`)

The `apihelper` package provides a bridge between CLI configuration and the API client:

```go
package apihelper

import (
    "context"
    "os"
    "path/filepath"
    "strings"

    "github.com/fclairamb/solidping/back/pkg/client"
    "github.com/fclairamb/solidping/back/pkg/cli/config"
)

// Helper manages API client creation and authentication
type Helper struct {
    config     *config.Config
    tokenPath  string
    apiClient  *client.SolidPingClient
}

// NewHelper creates a new API helper
func NewHelper(cfg *config.Config, tokenPath string) *Helper {
    return &Helper{
        config:    cfg,
        tokenPath: tokenPath,
    }
}

// GetClient returns an authenticated API client
func (h *Helper) GetClient(ctx context.Context) (*client.SolidPingClient, error) {
    if h.apiClient != nil {
        return h.apiClient, nil
    }

    token, err := h.resolveToken(ctx)
    if err != nil {
        return nil, err
    }

    cfg := client.Config{
        BaseURL: h.config.URL,
        Token:   token,
    }

    h.apiClient, err = client.New(cfg)
    return h.apiClient, err
}

// resolveToken gets token from file, PAT, or auto-login
func (h *Helper) resolveToken(ctx context.Context) (string, error) {
    // Priority 1: Token file
    if token, _ := readTokenFile(h.tokenPath); token != "" {
        return token, nil
    }

    // Priority 2: PAT from config
    if h.config.Auth.PAT != "" {
        return h.config.Auth.PAT, nil
    }

    // Priority 3: Auto-login with username/password
    if h.config.Auth.Username != "" && h.config.Auth.Password != "" {
        token, err := h.autoLogin(ctx)
        if err != nil {
            return "", err
        }
        // Save token for future use
        _ = saveTokenFile(h.tokenPath, token)
        return token, nil
    }

    return "", errors.New("no authentication configured")
}

// Login performs login and saves token
func (h *Helper) Login(ctx context.Context, org, username, password string) (string, error) {
    // Implementation as shown in previous example
    // ...
    return token, saveTokenFile(h.tokenPath, token)
}

// Logout removes saved token
func (h *Helper) Logout() error {
    return deleteTokenFile(h.tokenPath)
}
```

### Configuration Loading with Koanf
Example configuration loading:
```go
import (
    "github.com/knadh/koanf/v2"
    "github.com/knadh/koanf/parsers/json"
    "github.com/knadh/koanf/providers/file"
)

func LoadConfig(path string) (*Config, error) {
    k := koanf.New(".")

    // Load JSON config file
    if err := k.Load(file.Provider(path), json.Parser()); err != nil {
        return nil, err
    }

    config := &Config{}
    if err := k.Unmarshal("", config); err != nil {
        return nil, err
    }

    return config, nil
}

type Config struct {
    URL  string `koanf:"url"`
    Org  string `koanf:"org"`
    Auth struct {
        Username string `koanf:"username"`
        Password string `koanf:"password"`
        PAT      string `koanf:"pat"`
    } `koanf:"auth"`
}
```

**Note**: When loading config, apply defaults:
```go
config := &Config{
    Org: "default", // Set default before unmarshaling
}
if err := k.Unmarshal("", config); err != nil {
    return nil, err
}
// If org is empty string after unmarshal, ensure it's "default"
if config.Org == "" {
    config.Org = "default"
}
```

**Authentication Token Management:**
1. Check for stored token file first (`~/.config/solidping/token`)
2. If token file exists, use it to create client
3. If not, check for PAT in config â†’ use it to create client
4. If not, check for username/password in config â†’ auto-login, save token, create client

**Token File Operations:**
```go
// Read token from file
func readToken(path string) (string, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        if os.IsNotExist(err) {
            return "", nil // No token file
        }
        return "", err
    }
    return strings.TrimSpace(string(data)), nil
}

// Save token to file
func saveToken(path, token string) error {
    if err := os.MkdirAll(filepath.Dir(path), 0755); err != nil {
        return err
    }
    return os.WriteFile(path, []byte(token), 0600)
}

// Delete token file
func deleteToken(path string) error {
    err := os.Remove(path)
    if os.IsNotExist(err) {
        return nil // Already deleted
    }
    return err
}
```

**Login Implementation:**
```go
func login(ctx context.Context, baseURL, org, username, password string) (string, error) {
    // Create client without token for login
    cfg := client.Config{BaseURL: baseURL}
    apiClient, err := client.New(cfg)
    if err != nil {
        return "", err
    }

    // Call login endpoint
    loginReq := client.PostApiV1OrgsOrgAuthLoginJSONRequestBody{
        Username: username,
        Password: password,
    }

    resp, err := apiClient.PostApiV1OrgsOrgAuthLoginWithResponse(ctx, org, loginReq)
    if err != nil {
        return "", err
    }

    if resp.StatusCode() != 200 || resp.JSON200 == nil {
        return "", client.ErrAuthenticationFailed
    }

    // Return the JWT token
    return resp.JSON200.Token, nil
}
```

## Additional Considerations

### Future Enhancements (Not in Initial Implementation)
These features could be added later:
- Interactive mode for adding checks (prompt for each field)
- Watch mode: `sp results watch --check-slug google` (live updates)
- Config management commands: `sp config set url https://...`
- Bash/Zsh completion scripts
- Multiple config profiles: `sp --profile production checks list`
- Bulk operations: `sp checks add --file checks.yaml`
- Export results to CSV: `sp results list --format csv`

### Security Considerations
- Store config file with `0600` permissions (user read/write only)
- Store token file with `0600` permissions (user read/write only)
- Never log passwords or tokens, even in verbose mode
- Use `golang.org/x/term` for secure password input (no echo to terminal)
- Validate SSL certificates by default (allow `--insecure` flag to skip for development)
- Sanitize user input before sending to API
- Clear sensitive data from memory after use
- Warn users if config/token files have overly permissive permissions
- Support reading credentials from environment variables for CI/CD (with clear warnings about security)

### Performance Considerations
- Reuse HTTP connections with `http.Client`
- Implement request timeout to prevent hanging
- Cache check UID lookups when converting slugs (in-memory, per-command)
- Stream large result sets instead of loading all into memory
- Show progress indicator for slow operations (>2 seconds)

### UX Improvements
- Colorize output based on status (green for up, red for down)
  - Use `github.com/fatih/color` for terminal colors
  - Respect `NO_COLOR` environment variable
  - Disable colors when `--json` flag is used
- Show helpful error messages with next steps
- Add `--quiet` flag to suppress all output except errors
- Progress bar for bulk operations (optional enhancement)
- `go-pretty` already provides nice table formatting with borders

### Cross-Platform Support
- Test on Linux, macOS, and Windows
- Handle path separators correctly (`filepath.Join`)
- Use `os.UserConfigDir()` for config location on Windows
- Support both Unix line endings (`\n`) and Windows (`\r\n`)
