# CLI: Bulk Check Creation for Load Testing

## Overview
Add a `-nb` (number) flag to the `checks add` CLI command to create multiple identical checks in a single command. This enables easy load testing of the SolidPing system with many simultaneous checks.

## Use Case
Testing overall system performance under load by quickly creating hundreds or thousands of checks without writing scripts or making multiple API calls manually.

## Command Enhancement

### Current Command
```bash
solidping client checks add <url> [flags]
```

### New Flag
```bash
-nb, --number <count>    Number of checks to create (default: 1)
```

## Behavior

### With `-nb` flag specified:
1. Create `<count>` checks with the same configuration
2. Each check gets a unique slug and name by appending a sequential number
3. All checks share the same:
   - Type (http, ping, tcp, etc.)
   - Target URL/host (with optional `{nb}` pattern replacement)
   - Period/interval
   - Timeout settings
   - Other configuration

### URL Pattern Replacement
- **`{nb}` pattern**: The URL can contain `{nb}` which will be replaced with the current check number
- **Example**: `https://api-{nb}.example.com` becomes `https://api-1.example.com`, `https://api-2.example.com`, etc.
- **Use case**: Testing different servers, API endpoints, or subdomains with a single command
- **Note**: If no `{nb}` pattern is present, all checks will target the same URL

### Naming Pattern
- **Base slug**: If `--slug` is provided, use it as base. Otherwise, generate from URL
- **Generated slugs**: `{base-slug}-1`, `{base-slug}-2`, ..., `{base-slug}-N`
- **Base name**: If `--name` is provided, use it as base. Otherwise, use URL
- **Generated names**: `{base-name} #1`, `{base-name} #2`, ..., `{base-name} #N`

### Progress Reporting
- Show progress indicator when creating multiple checks
- Report success/failure count at the end
- In JSON mode, return array of created checks with success/error status for each

## Examples

### Create 100 HTTP checks to test Google
```bash
solidping client checks add https://www.google.com \
  --name "Google Load Test" \
  --slug "google-load" \
  --interval 1m \
  -nb 100
```

Creates:
- `google-load-1`, `google-load-2`, ..., `google-load-100`
- Names: "Google Load Test #1", "Google Load Test #2", ..., "Google Load Test #100"

### Create 500 ping checks
```bash
solidping client checks add 8.8.8.8 \
  --type ping \
  --name "DNS Performance Test" \
  -nb 500
```

### Create checks with {nb} pattern for different servers
```bash
solidping client checks add https://api-{nb}.example.com/health \
  --name "API Server Load Test" \
  --slug "api-server" \
  -nb 10
```

Creates checks targeting:
- `https://api-1.example.com/health`
- `https://api-2.example.com/health`
- ...
- `https://api-10.example.com/health`

### JSON output mode
```bash
solidping client checks add https://api.example.com \
  --json \
  -nb 10
```

Returns:
```json
{
  "total": 10,
  "created": 10,
  "failed": 0,
  "checks": [
    {
      "uid": "...",
      "slug": "api-example-com-1",
      "name": "https://api.example.com #1",
      "status": "created"
    },
    ...
  ]
}
```

## Implementation Details

### Error Handling
- If creation of any check fails, continue with remaining checks
- Report total success/failure count at the end
- Exit with error code if ANY check creation failed
- In JSON mode, include error details for each failed check

### Performance Considerations
- Create checks sequentially (not in parallel) to avoid overwhelming the API
- Consider adding a `--batch-size` flag in future for parallel creation with rate limiting
- Show progress indicator for long-running operations (more than 10 checks)

### Validation
- Validate that `-nb` is a positive integer (1 to 10,000)
- Warn if creating more than 1,000 checks (confirm with user prompt unless `--force` flag)
- Ensure generated slugs don't conflict with existing checks (API will reject duplicates)
