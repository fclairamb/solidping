# Authentication Implementation Specification

## Overview

This document outlines the implementation plan for authentication in SolidPing, inspired by the RealAssets project. The goal is to implement a complete authentication system while making minimal changes to the existing database structure.

## Current State Analysis

### What Already Exists in SolidPing

**Database Models** (`back/internal/db/models/models.go`):
- `User` - with UID, OrganizationUID, AuthProviderUID, UserID, PasswordHash, Role
- `UserToken` - with UID, Token, Type (pat/refresh), Properties, ExpiresAt
- `AuthProvider` - with UID, OrganizationUID, Type, Config

**Database Schema** (migrations already created):
- `users` table with password_hash, role (admin/user/viewer)
- `user_tokens` table with token, type (pat/refresh), expires_at
- `auth_providers` table for OAuth support

**OpenAPI Spec** (`back/internal/app/openapi/solidping.yaml`):
- `POST /api/v1/orgs/{org}/auth/login` - defined
- `POST /api/v1/orgs/{org}/auth/logout` - defined
- `GET /api/v1/orgs/{org}/auth/tokens` - defined
- `POST /api/v1/orgs/{org}/auth/tokens` - defined
- `DELETE /api/v1/orgs/{org}/auth/tokens/{token_uid}` - defined

**Database Service Interface** (`back/internal/db/service.go`):
- User CRUD operations already implemented
- UserToken CRUD operations already implemented

### What Needs to Be Implemented

1. **Password Utilities** - Argon2id hashing (copy from RealAssets)
2. **Auth Service** - Business logic for login, logout, token management
3. **Auth Handler** - HTTP handlers for auth endpoints
4. **Auth Middleware** - RequireAuth and RequireOrgAccess
5. **Base Handler** - Error codes and response utilities
6. **JWT Configuration** - Secret management in config
7. **Route Registration** - Wire handlers to routes

**Key Implementation Notes**:
- JWT tokens for short-lived access (1h default)
- Refresh tokens stored in `user_tokens` table with type='refresh'
- PAT tokens prefixed with `pat_` for easy identification
- PAT validation with 15-minute caching to reduce DB queries
- Support for both password hash verification and plaintext (dev only: `$plaintext$password`)

func (s *Server) setupRoutes() {
    // ... existing setup

    // Create auth service and handler
    authService := auth.NewService(s.dbService, s.config.Auth.JWTSecret)
    authHandler := auth.NewHandler(authService, s.config)
    authMiddleware := middleware.NewAuthMiddleware(authService, s.dbService, s.config)

    orgAuthProtected.DELETE("/tokens/:token_uid", authHandler.RevokeToken)

    // Protected API routes (example)

    // ... register other handlers with protection

## Database Changes Required

### Minimal Changes (Recommended)

No changes to existing schema are strictly required. The current structure supports:
- Password authentication via `users.password_hash`
- PAT tokens via `user_tokens` with `type='pat'`
- Refresh tokens via `user_tokens` with `type='refresh'`
- OAuth providers via `auth_providers` table

2. **Add `auth/me` endpoint** to OpenAPI spec (currently missing)

3. **Add `auth/refresh` endpoint** to OpenAPI spec (currently missing)

## Token Strategy

### Access Tokens (JWT)
- Short-lived (1 hour default)
- Stateless validation
- Contains: `user_uid`, `org_slug`, expiry
- Signed with HS256

### Refresh Tokens
- Long-lived (7 days default)
- Stored in `user_tokens` table with `type='refresh'`
- Random 32-byte hex string with timestamp
- Stored with auth context (user agent, IP) for audit

### Personal Access Tokens (PAT)
- Prefixed with `pat_` for identification
- Stored in `user_tokens` table with `type='pat'`
- User-defined expiry
- Cached for 15 minutes to reduce DB queries
- Properties field stores name, description

## Security Considerations

1. **Password Hashing**: Argon2id with secure defaults (64MB memory, 3 iterations)
2. **Constant-time Comparison**: For password verification
3. **Token Expiry**: Short-lived access tokens, longer refresh tokens
4. **Audit Trail**: Auth context stored with refresh tokens
5. **Soft Delete**: Tokens are soft-deleted, not hard-deleted
6. **Organization Scoping**: All tokens scoped to organization

## Summary

This implementation plan leverages the existing database structure in SolidPing while borrowing proven patterns from RealAssets. The key benefits are:

1. **No database schema changes required** for core functionality
2. **Proven patterns** from RealAssets for JWT + refresh token flow
3. **PAT support** for API access without exposing user credentials
4. **Multi-tenant ready** with organization scoping
5. **Secure defaults** with Argon2id and short-lived JWTs
